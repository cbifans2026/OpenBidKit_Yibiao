# 埋点统计功能完整方案

公告管理、资源管理不属于埋点统计功能，保持不变。其他功能完全按照下面写的来开发，没有提到的功能删除掉。
多标签页设计，每次进入标签页再取本页面数据，不要一次取所有的。

## 架构

### AE库
保持现有埋点统计采集数据不变，应该是够用的。
### D1库
长期存储统计数据，防止AE库自动清空而丢失历史数据。
可以完全放弃D1中关于埋点统计的历史数据和结构，基于新需求重新设计新版。
### 定时任务
每天凌晨2点执行，按业务需求把统计结果存入D1。

## 业务需求
### 概览改造

总客户端数、总打开次数、总页面访问量、总事件数、累计AI请求数：这些是存储在D1中的，直接读取；总客户端数包含 `/track` 实时写入的新客户端，其余累计数主要由定时任务汇总前一天AE数据。
定时任务：每天把前一天新增的事件计数加到总数中，并基于D1客户端表重算总客户端数。

今日新增客户端、7日新增客户端，从D1中取，今日活跃客户端，从AE库实时统计，不进D1库。

D1库存储所有客户端ID，在计算今日新增、7日新增时候，可以直接从D1库进行比对。`/track` 是客户端埋点上报接口，客户端打开软件、访问页面、上报配置、发起AI请求或点击资源时都会调用它；`/track` 会先写入AE，仅当事件中带有 `client_id` 且 `client_created_at` 距当前业务日期不超过3天时，才尝试把该客户端实时写入D1并记录初次访问时间。

GitHub Stars、fork、issue/PR、打开仓库保持现功能不变。

去掉时间范围选择，现在概览中已经用不到了。

每日统计：显示每日活跃客户端数、软件打开次数、页面访问量。只展示10天的数据，当天数据从AE取，之前数据从D1取。定时任务也是把每天的统计结果直接存入D1库。

留存概览逻辑不变，从AE中计算，不进行持久化。

### 新增“客户端统计”

展示所有客户端id、首次访问时间、访问天数、最近活跃日期、最近活跃版本。
数据从D1取，可能包含 `/track` 实时写入的近3天新客户端；访问天数、最近活跃日期、最近活跃版本由定时任务每天统计前一天结果后写入D1。
最近活跃日期只看前一天有没有活跃就行，不需要每次活跃都更新。

每行数据有一个操作：“详情”，点击弹窗先选择时间维度：7天、30天、全部，然后展示该客户端活跃具体日期、每天的事件数，每类事件数，从AE库直接分析，AE清了就清了，不持久化。

### 访问分析改造

时间范围可选“历史、今天、7天、30天”，默认历史，直接从D1取，忽略AE库。
选择其他时间维度，从AE计算，忽略D1库。
历史范围中的页面/版本客户端数通过D1维度客户端关系表长期去重计算，不按每日客户端数累加。
优化版本分布排序逻辑，应该按版本号倒排，现在有个问题是2.10.1是大于2.9.1的，但是现在的排序逻辑导致2.9.1显示在2.10.1的上面了，需要优化排序算法。

### 配置使用改造

时间范围可选“历史、今天、7天、30天”，默认历史，直接从D1取，忽略AE库。
选择其他时间维度，从AE计算，忽略D1库。
定时任务持久化前一天数据至D1，新项则新增；次数可以累加，客户端数通过D1维度客户端关系表长期去重后重算，不能按每日去重数累加。

### 模型使用改造

时间范围可选“历史、今天、7天、30天”，默认历史，直接从D1取，忽略AE库。
选择其他时间维度，从AE计算，忽略D1库。
定时任务持久化前一天数据至D1，新项则新增；请求数和Token可以累加，客户端数通过D1维度客户端关系表长期去重后重算，不能按每日去重数累加。
增加查询条件：服务商、域名、模型，都使用可搜索的下拉框实现。

### 最近事件改造

增加查询条件：事件，使用可搜索的下拉框。
最近事件数据只在AE库，不入D1。
